## 三条主线工作流

第二轮实际做了三件事，不只是原计划的"建 api-client"：

### 主线 1：死代码清理（先做）

第一轮完成后回头审计 admin 的 raw fetch 调用，发现有 17 个 fetch 调用打到 admin 自己根本不存在的 `/api/*` 路径（不是 `/api/admin/*`）—— 因为 admin 没配 rewrites，这些请求全是 silent 404。

进一步审计这些 endpoint 是否真的是活路由后发现：

| 模块 | 处理决策 | 原因 |
|---|---|---|
| reports | **整套删除** | 业务上已废弃；admin route + page + core/types/report.ts 全删 |
| SupportTicketsTab | **admin 端删除，apps/api 端保留** | 工单已改为 email 处理，但 apps/api/support-ticket route 是用户提交工单的活入口（POST 写表 + Resend 发邮件） |
| /admin/book-orders | **admin 端整页删除** | 被 OrdersTab 取代；但 apps/api/book-orders 是另一回事，业务关系不明（见 backlog B7） |

**关键判断**：删除 admin 端 UI 不等于可以删除 apps/api 的同名 route。**用户侧路由必须用业务证据保留**，宁可暂留登记 backlog 也不基于猜测删除。

### 主线 2：配置 & 基础设施

清理完死代码，活路由的 cross-app proxy 配置全面缺失。3 个 endpoint 必须从 admin 域名转发到 apps/api 域名：

- `/api/orders/*` → OrdersTab
- `/api/force-update/*` → OverviewTab
- `/api/generate-abc-cards/*` → CardsTab real 子 tab

实施过程中踩了 3 个坑：

**坑 1**：本地 dev 无法验证 rewrites（中国 DNS 把 vercel.app 解析到 Facebook IP，curl/ping 全部 timeout）→ 改走 production 直接验证

**坑 2**：production 部署后仍然 404 + `DNS_HOSTNAME_RESOLVED_PRIVATE` → 排查发现 Vercel 上配的环境变量没传给 next.config.js 的 build process → 真正原因是 **Turborepo 默认隔离环境变量**，turbo.json 的 build/dev 任务必须显式声明 env

**坑 3**：environment variables 改动后做的 redeploy 是用旧 build cache → 必须用空 commit 强制 fresh build

**最终配置**：
- `apps/admin/next.config.js`：6 条 rewrite 规则（3 个 endpoint × 根路径 + 带参数）
- `apps/admin/.env.local.example`：documentation
- `turbo.json`：build/dev 任务声明 5 个环境变量
- Vercel admin 项目：`NOVAME_API_URL=https://novame-monorepo-api.vercel.app`

### 主线 3：packages/api-client（原计划主任务）

四个设计决策落地：

1. **方法粒度**：分层 API（通用 `request()` + `get/post/patch/put/delete` 便利方法），不为还没出现的 mobile 消费者预先设计 endpoint-specific 方法
2. **错误处理**：所有失败抛 `ApiError`（kind: 'http' | 'network'，附 status + body + cause），调用方 try/catch
3. **构造参数**：`{ baseUrl, getToken? }`；admin 用 `baseUrl: ''`（同源）+ `getToken: () => null`（cookie session），mobile 阶段 2.3 注入 Supabase access token
4. **包结构**：source-only export（main 指向 src/index.ts），跟 packages/core / packages/ui-tokens 一致

**关键实现细节**：
- FormData 自动检测：避免对 multipart 内容做 JSON.stringify，让 fetch 自己设置 Content-Type 边界
- HTML 404 页面安全解析：JSON.parse 包裹 try-catch，body 退回 string，不再炸 SyntaxError（解决了 RealUsersTab 一类 silent failure 问题）
- TS override modifier：ApiError.cause 必须用 `override cause?: Error`（Error 基类的 cause 字段）

**admin 38 处 raw fetch 收编**：
- 4 批分文件迁移（5A: PostsTab / 5B-1, 5B-2, 5B-3 各 3 文件）
- 每批用 Python 脚本（heredoc 单引号 EOF + str.replace）做精确字符串替换，避开 perl/sed 的转义陷阱
- 每批跑 `pnpm --filter @novame/admin type-check` 验证
- 净删 89 行 boilerplate（`headers: { 'Content-Type' }`, `JSON.stringify`, `await res.json()`, `if (!res.ok)`）

**DELETE 方法的 endpoint 兼容性**：
- /api/admin/wisdoms：兼容 querystring + body —— 用 querystring
- /api/admin/announcements：querystring 形式
- /api/admin/default-users + default-cards：仅支持 body —— 用 body
- /api/admin/seek-questions：所有 mutation 都通过 POST + `action: 'delete'|'remove_card'|...` dispatch，没有真 DELETE handler

为支持上述差异，ApiClient.delete 签名扩展为 `(path, body?, options?)`。

## 经验教训

1. **死代码清理优先级高于新建**：1.4 第一轮跳过了死代码清理，导致第二轮发现 17 处 silent 404 + 1 处潜在数据流分裂（B7）。下次新阶段开始前，先扫一遍上一阶段未覆盖的代码路径。

2. **Turborepo env 声明是 monorepo 必修课**：next.config.js 用 `process.env.X` 时，X 必须在 turbo.json 的 build/dev 任务里声明，否则 Vercel 上配了变量也读不到。这条 mobile 阶段 2 同样适用。

3. **DELETE 方法的 querystring vs body 不通用**：每个 route 的 handler 怎么读 id 必须真去看 source code，不能假定。ApiClient 的 delete 方法支持两种形式，让 admin 收编时能保持原 endpoint 行为。

4. **shell 字符串处理选择题**：本轮经历了 perl 多次失败（`@` 符号、嵌套大括号、中文 unicode、换行嵌套），最终稳定方案是 **Python heredoc + 单引号 EOF + str.replace**，对所有特殊字符免疫。下一阶段沿用。

5. **MA cosmetic check 不阻塞 production 时不修**：B9 Vercel TypeCheck failure 看不到详细日志，但 production 实际部署成功。区分"装饰性失败"和"实际阻塞"很重要。
